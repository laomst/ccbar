# CCBar 新特性开发流程

## 概述

本 skill 定义了 CCBar 项目新特性开发的标准流程，确保文档同步更新、用户影响得到评估。

## 开发流程

### 步骤 1：创建特性规格文档

1. 确定特性编号（如 `story-01`、`story-02`）
2. 创建目录：`docs/specs/story-xx/`
3. 创建规格文档：`doc0.spec.md`

**规格文档结构：**

```markdown
# Story-XX: 特性名称 - 需求分析文档

## 1. 需求背景
### 1.1 当前实现
### 1.2 用户需求

## 2. 需求分析
### 2.1 布局结构
### 2.2 交互行为
### 2.3 视觉规范

## 3. 技术方案
### 3.1 UI 组件选型
### 3.2 状态管理
### 3.3 事件处理

## 4. 影响范围
### 4.1 需修改的文件
### 4.2 不受影响的部分

## 5. 验收标准
### 5.1 功能验收
### 5.2 UI 验收
### 5.3 兼容性验收

## 6. 风险评估

## 7. 后续优化建议
```

参考模板：[spec-template.md](assets/spec-template.md)

### 步骤 2：更新整体功能描述

同步更新 `docs/spec.md` 中相关章节：

| 规格文档章节 | spec.md 对应章节 |
|-------------|-----------------|
| 需求背景 | 3. 核心功能需求 |
| 交互行为 | 3. 核心功能需求 / 4. 功能示意图 |
| 数据结构 | 5. 配置数据结构 |
| UI 变化 | 6. 配置界面设计 |

**更新检查清单：**
- [ ] 功能需求章节是否需要补充
- [ ] 示意图是否需要更新
- [ ] 数据结构是否需要修改
- [ ] 配置界面描述是否需要调整

### 步骤 3：评估用户影响

**评估维度：**

| 维度 | 说明 | 需要 README 更新 |
|------|------|-----------------|
| 使用方式变化 | 用户操作流程改变 | ✅ 是 |
| 界面变化 | UI 布局或样式调整 | ✅ 是 |
| 新增配置项 | 设置面板新增选项 | ✅ 是 |
| 仅内部优化 | 性能、代码重构 | ❌ 否 |
| Bug 修复 | 不影响使用习惯 | ❌ 否 |

**README 更新内容：**

- 功能特性列表
- 使用方式说明
- 命令生成规则（如有变化）
- 示意图（如有）

### 步骤 4：确认与实现

1. 向用户确认规格文档内容
2. 获得确认后开始实现
3. 实现完成后更新验收状态

## 快速参考

### 文件路径

```
docs/
├── spec.md              # 整体功能描述（需同步更新）
├── specs/
│   └── story-xx/        # 特性规格目录
│       └── doc0.spec.md # 特性规格文档
README.md                # 项目说明（用户影响时更新）
```

### 触发条件

- 用户明确要求开发新特性
- 需要添加新功能
- 需要修改现有功能行为
- 需要规划功能实现方案
